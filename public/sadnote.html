Chapter 1: Problem Analysis (Fishbone)

Definition: Also known as the Ishikawa Diagram or Cause-and-Effect Diagram. It is exclusively used in the Analysis Phase to understand the "As-Is" (current) problem before we design a solution.

1.1 The "Grouping" Methodology (Lecture Oct 30)

You cannot just draw the fishbone immediately. The lecture emphasized a specific pre-processing step called "Grouping" to organize your thoughts. Without this, your diagram will be messy and illogical.

"We don't draw just for fun. We draw to communicate with ourselves first. Grouping reveals the structure of the problem." - Lecture Oct 30
  1. Brainstorming: Write down every possible cause messily on the board. (e.g., "Rain", "Lazy", "Sick", "No Bus", "Traffic Jam", "Played Games").
  2. Grouping: Draw circles around related items to create logical clusters.
    • (Traffic Jam, No Bus) → Group Name: Traffic
    • (Rain, Hot Weather) → Group Name: Weather
    • (Lazy, Played Games) → Group Name: Habit
    • (Sick, Stomach Ache) → Group Name: Health
  3. Drawing: Only after grouping do you draw the skeleton with main bones corresponding to your groups.

1.2 The "6Ms" Framework (Standard Categories)

While the lecture used specific examples like "Traffic" and "Weather", in formal exams you are often expected to use standard manufacturing/business categories (The 6Ms) to ensure you have covered all bases.

  • Man (People): Staff skills, training, fatigue, lack of motivation.
  • Machine: Equipment failures, computers, tools, software glitches.
  • Material: Raw supplies, information quality, paperwork, parts.
  • Method: Rules, policies, procedures, workflows, regulations.
  • Measurement: Data accuracy, inspections, calibration of tools.
  • Mother Nature (Environment): Noise, heat, layout, lighting, humidity.

1.3 The "5 Whys" Technique

The Fishbone shows the main "Bone," but how do you find the deep sub-bones? The "5 Whys" is the standard technique for finding the root cause.

Example: Late to Class
1. Why are you late? Traffic.
2. Why was traffic bad? I left the house late.
3. Why did you leave late? Alarm didn't go off.
4. Why didn't it go off? Battery died.
5. Why? I forgot to check the maintenance schedule.
Root Cause: Maintenance/Method (Not Traffic).

1.5 Managing Expectations (Exam Essay Point)

Why do we draw this? It is a negotiation tool. If you propose a "Smart Alarm Clock" app, the Fishbone diagram proves to the client that your app only solves the "Habit" bone. It does not solve the "Traffic" or "Weather" bones. This protects you from clients saying "Why am I still late?" after buying your app.

Chapter 2: Process Modeling (Activity Diagram)

Used to describe Business Processes (As-Is or To-Be). It shows the Sequence of actions.

2.1 The "Indeterministic" Trap (The '5' Problem)

CRITICAL: Never split an arrow without a Decision Diamond.

The Scenario: You have an Input A (Number). You want to branch based on value.

The Mistake: Drawing two arrows directly from Activity A. One labeled [x < 5] and one labeled [x > 5].

The Crash: If the input is exactly 5, the system crashes because it matches neither path (or both, depending on labels). This is an "Indeterministic System".

The Solution: Always use a Diamond (Decision Node). The diamond forces you to define Mutually Exclusive paths (e.g., [x <= 5] vs [x > 5]).

2.2 Concurrency (Fork & Join)

The notes so far showed sequential flow. Real systems do things in parallel. 

      graph TD
        Start((Start)) --> Fork[==== FORK ====]
        style Fork fill:#000,stroke:#000,color:#fff,height:5px
        
        Fork --> A[Print Receipt]
        Fork --> B[Open Gate]
        
        A --> Join[==== JOIN ====]
        B --> Join
        style Join fill:#000,stroke:#000,color:#fff,height:5px
        
        Join --> End((End))
Fig 2.1: Fork and Join Nodes
  • Fork (Black Bar, 1 In, Multiple Out): Splits flow into parallel threads. (e.g., "Print Receipt" AND "Open Gate" happen at the same time).
  • Join (Black Bar, Multiple In, 1 Out): Waits for ALL parallel threads to finish before moving on.

2.3 Object Nodes

Activities are Verbs. Sometimes you need to show the Data (Noun) moving between them.

Notation: A rectangle (not rounded) placed between activities. 

      flowchart LR
        A(Fill Form) --> B[Application Form]
        B --> C(Review Form)
Fig 2.2: Object Node Notation

Chapter 3: Functional Modeling (Use Case)

Describes Functional Requirements. "Who uses the system to do WHAT?"

3.1 The "Purpose" Concept

A Use Case is a user's goal, not a small step. "Log In" is a Use Case, but "Click Button" is NOT.

3.2 Relationships: The "OTP Story" (Nov 13)

Why do we use << include >>? The lecturer gave a specific example.

  • Scenario: You have 20 Use Cases (Submit HW, Grade HW, View Report...). All of them require Login.
  • Bad Design: You write the "Login" steps inside all 20 descriptions.

Bad Design (Duplication) 

      graph TD
        User((User))
        subgraph System
        A(Submit HW\nIncludes Login Steps)
        B(Grade HW\nIncludes Login Steps)
        C(View Report\nIncludes Login Steps)
        end
        User --> A
        User --> B
        User --> C
Fig 3.1: Bad Design (Duplication)
  • The Problem: The system changes from "Password" to "OTP" (One Time Password). You now have to edit 20 documents.
  • The Solution (Include): Extract "Login" as a separate Use Case. The other 20 Use Cases point to it. Now you only update the "Login" description once.

Good Design (Reuse) 

      graph TD
        User((User))
  subgraph System
    A(Submit HW)
    B(Grade HW)
    C(View Report)
    L(Login)
  end
  User --> A
  User --> B
  User --> C
  A -.->|"< include >"| L
  B -.->|"< include >"| L
  C -.->|"< include >"| L
Fig 3.2: Good Design (Include)

3.3 Include vs Extend (Crucial Exam Distinction)

3.3.1 Include Relationship

Logic: Mandatory. The base case cannot finish without executing the included case. It is like a subroutine call that happens 100% of the time.

Example: "Withdraw Cash" includes "Verify PIN". (You cannot withdraw without PIN). 

      graph LR
        A(Withdraw Cash) -.->|"<< include >>"| B(Verify PIN)
Fig 3.3: Include Relationship

3.3.2 Extend Relationship

Logic: Optional. The base case works fine on its own. The extension only runs if a specific condition is met.

Example: "Withdraw Cash" extended by "Print Receipt". (User might say No). 

      graph LR
        B(Print Receipt) -.->|"<< extend >>"| A(Withdraw Cash)
Fig 3.4: Extend Relationship

3.4 Generalization (Inheritance)

Just like classes, Actors and Use Cases can inherit properties.

  • Actor Generalization: A "Manager" is a child of "Staff". The Manager can do everything "Staff" can do, plus more. Draw an arrow with a hollow triangle from Manager pointing TO Staff. 
      classDiagram
        class Staff
        class Manager
        Staff <|-- Manager
Fig 3.5: Actor Generalization
  • Use Case Generalization: "Pay Bill" is the parent. "Pay by Credit Card" and "Pay by Cash" are children that inherit the general behavior. 
      classDiagram
        class PayBill
        class CreditCard
        class Cash
        PayBill <|-- CreditCard
        PayBill <|-- Cash
Fig 3.6: Use Case Generalization

3.5 The System Boundary

The notes mention "System" in text, but in the diagram, you must draw a Box around the Use Cases.

Rule: Actors go OUTSIDE the box. Use Cases go INSIDE the box. This visualizes the scope of your software. 
      graph TD
        User((User))
        subgraph "System Boundary (The Box)"
        A(Login)
        B(View Dashboard)
        end
        User --> A
        User --> B
Fig 3.7: System Boundary

3.6 Use Case Scenarios (The Textual Part)

The diagram is only the summary. The "Meat" is the textual description.

  • Pre-condition: What must be true before starting? (e.g., User is logged in).
  • Post-condition: What is true after finishing? (e.g., Database is updated).
  • Main Success Scenario: The "Happy Path" steps (1, 2, 3...).
  • Alternative Flows: What happens if something goes wrong (errors)?

Visualizing the Scenario (Sequence Diagram) 

      sequenceDiagram
        actor User
        participant System
        participant Database
        
        Note over User, System: Pre-condition: User is logged in
        
        User->>System: 1. Request Report
        System->>Database: 2. Query Data
        Database-->>System: 3. Return Data
        System-->>User: 4. Display Report
        
        Note over User, System: Post-condition: Report is displayed

Chapter 4: Data Flow Diagrams (DFD)

Describes the Logical Flow of data. It does NOT show physical actions (like "walking to the desk").

4.1 The 4 Critical Rules (Exam Traps)

1. No Miracles: Process with output but NO input. (Impossible to create data from thin air).
2. No Black Holes: Process with input but NO output. (Useless process).
3. No Entity-to-Entity: External entities cannot talk directly. (Out of scope).
4. No Entity-to-Store: Entities cannot touch the database directly (Must go through a process).

1. Miracle (No Input) 

        flowchart TD
          P1((Process)) -- Data --> E1[Entity]
Fig 4.1a: Miracle (Incorrect)

Why it is wrong: A process cannot generate data from nothing. It must have at least one input arrow.

2. Black Hole (No Output) 

        flowchart TD
          E2[Entity] -- Data --> P2((Process))
Fig 4.1b: Black Hole (Incorrect)

Why it is wrong: A process that takes data but produces nothing is useless. It must have at least one output arrow.

3. Entity-to-Entity (Direct) 

        flowchart LR
          E3[Entity A] -- Data --> E4[Entity B]
Fig 4.1c: Entity-to-Entity (Incorrect)

Why it is wrong: DFD models the SYSTEM. If two external people talk to each other without using the system, it is out of scope.

4. Entity-to-Store (Direct) 

        flowchart LR
          E5[Entity] -- Data --> D1[(Data Store)]
Fig 4.1d: Entity-to-Store (Incorrect)

Why it is wrong: External entities cannot magically teleport data into the database. They must use a system process (like "Save Data") to do it.

4.2 Context Diagram vs. Level 0

The notes often jump to Diagram 0, but you must know the difference.

  • Context Diagram: Has only ONE Process box (labeled "0: The System"). It shows all external entities and main data flows. NO Data Stores are usually shown here. 
      flowchart LR
        E1[Customer]
        P0((0. Food Ordering System))
        E2[Kitchen]
        E1 -- Order --> P0
        P0 -- Receipt --> E1
        P0 -- Ticket --> E2
Fig 4.2: Context Diagram
  • Level 0: This is the "Zoom in" version. It breaks the one big process into major sub-processes (1.0, 2.0, 3.0). Data Stores appear here. 
      flowchart LR
        E1[Customer]
        P1((1.0 Receive Order))
        P2((2.0 Send to Kitchen))
        D1[(Orders DB)]
        E2[Kitchen]
        
        E1 -- Order --> P1
        P1 -- Receipt --> E1
        P1 -- Order Details --> D1
        D1 -- Order Details --> P2
        P2 -- Ticket --> E2
Fig 4.3: Level 0 Diagram

4.3 Balancing (Consistency)

This is a major exam check. The inputs and outputs on the Context Diagram must match the inputs and outputs on the Level 0 Diagram exactly. You cannot suddenly add a new data flow from an external entity in Level 0 that wasn't in the Context Diagram. 

      flowchart TD
        subgraph "Context Diagram"
        E1[Entity] -- Input A --> P0((System))
        P0 -- Output B --> E1
        end
        
        subgraph "Level 0 Diagram (Balanced)"
        E2[Entity] -- Input A --> P1((1.0))
        P1 -- Internal Data --> P2((2.0))
        P2 -- Output B --> E2
        end
Fig 4.4: Balancing Example

Notice: "Input A" and "Output B" exist in BOTH layers. This is Balanced.

4.4 The Data Dictionary

DFD arrows only show the "packet" name. A Data Dictionary explains what is inside. 

      classDiagram
        class PatientInfo {
          +String Name
          +String Address
          +String Phone
          +Date DOB
        }
Fig 4.5: Data Dictionary Definition

Note: This defines the data structure inside the "Patient Info" flow arrow.

Chapter 5: The "Fragment Method" (Advanced DFD)

This technique from the Nov 27 lecture explains how to build a complex DFD from scratch.

5.1 The Workflow

  1. Start with Use Cases: Take one Use Case (e.g., "Create Order"). 
              graph LR
            Actor((User)) -- Uses --> UC(Create Order)
    Fig 5.1: Step 1 - Identify Use Case
  2. Create a Fragment: Turn that Use Case into a single DFD Process.
    • Actor → External Entity.
    • Use Case → Process Box.
    • Association → Data Flow Arrows (Input/Output). 
              flowchart LR
            E[User] -- Order Details --> P((Create Order))
            P -- Confirmation --> E
    Fig 5.2: Step 2 - Create Process Fragment
  3. Add Data Stores (The "Missing" Link): Use Cases don't show data stores. You must ask: "Does this process need to remember data?" If yes, add a Data Store. 
              flowchart LR
            E[User] -- Order Details --> P((Create Order))
            P -- Save Order --> D[(Orders DB)]
    Fig 5.3: Step 3 - Add Data Store
  4. Combine: Merge all fragments into Diagram 0.

Part 1: Short Essay Questions (Theory & Justification)

Q1. The "Indeterministic" Trap (Activity Diagrams)
Explain why specifically defining conditions in a Decision Diamond is critical. Use example input X = 10 where branches are [X < 10] and [X > 10].
Show Answer

Answer: Defining precise conditions is critical for system stability because indeterminate logic causes system crashes. In the example where X=10, the branches [X < 10] and [X > 10] fail to cover the input value. The system does not know which path to take, leading to a hang or crash. This is called a "Non-Mutually Exclusive" or "Indeterministic" error. A correct design must cover all possibilities, for example using [X <= 10].

Q2. Managing Client Expectations (Fishbone)
You are building a "Diet Tracking App" for a client to solve "Rising Obesity Rates." How do you use Fishbone to limit scope?
Show Answer

Answer: The Fishbone diagram helps manage expectations by visually separating root causes into categories. While "Rising Obesity" is the main problem, the app can only address specific "Bones" like Method (Tracking calories) or Measurement (Weight logging). The diagram proves to the client that causes under Man (Willpower), Mother Nature (Food availability), or Health (Genetics) are outside the software's control, preventing the client from blaming the app for user failure.

Q3. Data Flow Integrity (DFD Rules)
Define "Miracle" and "Black Hole". Why must a process produce output?
Show Answer

Answer: A "Miracle" is a process that creates output with no input (data from thin air). A "Black Hole" is a process that takes input but produces no output. A process MUST produce output because, in an information system, if data enters a process and absolutely nothing happens to it or leaves it, the process served no purpose and is redundant.

Q4. Use Case Relationships
Explain the difference between << extend >> and << include >> regarding execution logic.
Show Answer

Answer: The fundamental difference is necessity. The << include >> relationship is mandatory; the base use case is incomplete without the included use case and will fail if it doesn't run. The << extend >> relationship is optional/conditional; the base use case is complete on its own, and the extension only runs if a specific condition is met.

Example: "Cook Pasta" << include >> "Boil Water" (Mandatory). "Cook Pasta" << extend >> "Add Spicy Sauce" (Optional).

Q6. Activity Diagram (Fork & Join)
Scenario: "Order Fulfillment". When approved, Print Packing Slip AND Send Email simultaneously. Order cannot complete until BOTH are done.
Show Diagram Answer 
        graph TD
          Start((Start)) --> Approve[Approve Order]
          Approve --> Fork[==== FORK ====]
          Fork --> Print[Print Packing Slip]
          Fork --> Email[Send Email]
          Print --> Join[==== JOIN ====]
          Email --> Join
          Join --> Complete[Mark Completed]
          Complete --> End((End))
Fig 6.2: Activity Diagram with Fork/Join

Explanation: Use a Heavy Black Bar (Fork) to split the flow. This indicates true parallelism. Both paths must merge into another Heavy Black Bar (Join) which acts as a "Wait" state. The process cannot proceed to "Mark Completed" until both tokens arrive at the Join.

Q7. Use Case Diagram (Generalization & Include)
Scenario: Library. Students and Professors borrow books. "Borrow" always requires "Check Fines". Professors can "Request Inter-library Loan".
Show Diagram Answer 
        graph TD
          Student((Student))
          Prof((Professor))
          
          %% Generalization: Professor IS A Student (in terms of capability)
          Prof -- "is a" --> Student

          subgraph "Library System"
            Borrow(Borrow Books)
            Check(Check Fines)
            Loan(Request Inter-library Loan)
          end

          Student --- Borrow
          Borrow -.->|"<< include >>"| Check
          
          %% Only Prof can do this
          Prof --- Loan
Fig 6.3: Use Case Generalization & Include

Explanation: The Professor inherits from Student (triangle arrow), so we don't need to draw the line to "Borrow" again for the Professor. "Borrow" includes "Check Fines" because it is mandatory.
